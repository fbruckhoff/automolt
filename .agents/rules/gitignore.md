Use this rule for any `.gitignore` file to maintain it clean and relevant.

## Git ignore maintenance principles

### Question every entry
For every line/pattern in a `.gitignore` file, ask:
- "Is this relevant to this specific project?"
- "Analyze the codebase, delete if not"

### Relevance analysis process
1. **Search for actual files/patterns** - Use `find_by_name` to check if ignored files exist
2. **Search for usage references** - Use `grep_search` to find tool/framework usage in source code
3. **Check project configuration** - Examine configuration files for tool references
4. **Verify dependency management** - Check what package manager the project actually uses

### Organization categories
Organize `.gitignore` entries in this order:

1. **Language core** - Byte-compiled files, object files, language-specific artifacts
2. **Distribution/packaging** - Build artifacts, distribution files, package files
3. **Dependency management** - Lock files, package manager configs
4. **Environments** - Virtual environments, environment variables, config files
5. **IDEs** - Editor configuration files
6. **OS-specific** - System-generated files
7. **Tool caches** - Linter, formatter, tool-specific caches
8. **Project-specific** - Local development files, scratch directories

### Specific tool guidance

#### Keep entries for:
- **Language artifacts** - Files automatically generated by the language (bytecode, object files, etc.)
- **Build artifacts** - Files generated by build tools and package managers
- **Virtual environments** - Development environment directories
- **Tool caches** - Files generated by tools actually used in the project
- **IDE configs** - Preventative measure for developers using different IDEs

#### Remove entries for:
- **Unused frameworks** - Frameworks not used in the project
- **Unused package managers** - Package managers not used for dependency management
- **Unused testing tools** - Testing frameworks when no tests exist
- **Unused documentation tools** - Documentation generators not used
- **Unused translation tools** - Internationalization tools when not internationalized

### Consolidation principles
- Group related entries under appropriate categories
- Remove duplicate entries
- Use concise, descriptive comments
- Keep entries in logical order

### Verification commands
Use these commands to verify relevance:
```bash
# Find actual files
find_by_name -pattern "<pattern>" -search_directory "<project_root>"

# Search for usage in source code
grep_search -includes "*.<extension>" -query "<tool_name>" -search_path "<project_root>"

# Check configuration files
read_file -file_path "<project_root>/<config_file>"
```

### Final checklist
Before completing `.gitignore` maintenance:
- [ ] Every entry has been questioned for relevance
- [ ] Entries are organized by category
- [ ] No duplicate entries exist
- [ ] Comments are concise and accurate
- [ ] Project-specific tools are included
- [ ] Unused tools are removed
- [ ] File is clean and readable
