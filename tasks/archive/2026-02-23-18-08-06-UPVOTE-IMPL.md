# UPVOTE-IMPL: Automation Upvote Policy + `moltbook upvote` Command

## 1) Objective

Implement upvote support in two surfaces:

1. **Automation action stage** must return a structured boolean field `upvote` and execute an upvote against the item it autonomously acted on when `upvote == true`.
2. **New top-level CLI command** `moltbook upvote` must support upvoting both posts and comments using either IDs or Moltbook URLs.

This spec also updates setup UX text so `BEHAVIOR.md` explicitly covers:

- how the agent posts/comments,
- and under what circumstances it upvotes.

---

## 2) Scope and Policy Constraints

### 2.1 In scope

- Action-stage structured output contract change (`ActionPlan` adds `upvote: bool`).
- Automation runloop upvote execution on autonomously acted items.
- Setup prompt copy updates for `BEHAVIOR.md` guidance.
- New `moltbook upvote` command with `--handle` and target parsing.
- API client + service support for post/comment upvote endpoints.
- Test coverage and documentation updates.

### 2.2 Explicit policy rules

1. **No automation downvotes.**
2. **No automatic downvote behavior in setup guidance.**
3. **Automation upvotes only for items it autonomously acts on in that cycle.**
4. **Automation must never upvote unrelated content discovered but not acted on.**

---

## 3) Current Baseline (Verified)

1. Action-stage model currently contains only:
   - `should_reply`, `reply_content`, `reason`.
2. Automation action stage currently posts comments only and never votes.
3. Setup `PROMPT_DESCRIPTIONS["behavior"]` talks about reply style only (no upvote guidance).
4. No top-level `upvote` command exists.
5. `MoltbookClient` currently has comment and search endpoints, but no vote endpoints.

---

## 4) External API Contract (Moltbook)

Per Moltbook API skill references:

- **Upvote post:** `POST /api/v1/posts/{post_id}/upvote`
- **Upvote comment:** `POST /api/v1/comments/{comment_id}/upvote`

Implementation must continue using:

- base URL `https://www.moltbook.com/api/v1`
- auth header `Authorization: Bearer <api_key>`

No `/downvote` automation behavior is introduced.

## 6) Automation Runloop Behavior Changes

File: `moltbook/services/automation_service.py`

## 6.1 Upvote target resolution

Define deterministic mapping for the acted item:

- if queue item type is `post` -> upvote target is that post (`post_id` resolved from queue item)
- if queue item type is `comment` -> upvote target is that comment (`item_id`)

This maps directly to “the item the action stage is posting or commenting on.”

## 6.2 Execution timing

Apply upvote only in the branch where the automation action is considered acted:

1. action stage produced `should_reply=true`
2. reply content is non-empty and normalized
3. for live mode: comment post call attempted/succeeded
4. then, if `action_plan.upvote` is true, attempt upvote on action target

### Important

- If `should_reply=false`, **ignore upvote request** and do not vote.
- If action stage fails / no action taken, do not vote.

## 6.3 Dry mode behavior (`automation start --dry`)

Dry mode must remain side-effect free:

- no comment writes,
- no upvote writes.

If `upvote=true` in dry mode, include “would upvote” information in observer/monitor output payload, but do not call API.

## 6.4 Failure semantics

Upvote failure should **not erase successful action state**:

- If reply/comment succeeded but upvote fails, preserve acted state (`replied_item_id` remains set).
- Log warning with enough context (handle, item_id, target_type, target_id).
- Continue cycle completion normally.

Rationale: reply side-effect is already committed and cannot be rolled back.

## 6.5 Optional observability extension (recommended)

Add one additional heartbeat event type for live/dry visibility:

- `ACTION_UPVOTE` (or split into `ACTION_UPVOTE_DRY_RUN` and `ACTION_UPVOTED`)

Payload fields should include:

- target type (`post`/`comment`)
- target ID
- dry/live flag
- API message (live, when available)

This is recommended for operator confidence but not strictly required to satisfy core behavior.

## 8) New CLI Command: `moltbook upvote`

## 8.1 Command surface

Create: `moltbook/commands/upvote_command.py`

Register in: `moltbook/cli.py`

Proposed signature:

```bash
moltbook upvote [OPTIONS] TARGET
```

Options:

- `--handle TEXT` (session-aware semantics, same as other agent commands)
- `--type [auto|post|comment]` (default: `auto`)

Argument:

- `TARGET`: post/comment ID or Moltbook URL

## 8.2 Target parsing contract

Support these inputs:

1. **Post URL**
   - `https://www.moltbook.com/posts/<post_id>`
   - resolves to post upvote

2. **Comment URL**
   - `https://www.moltbook.com/posts/<post_id>#comment-<comment_id>`
   - resolves to comment upvote

3. **Raw ID with explicit type**
   - `--type post <id>`
   - `--type comment <id>`

4. **Raw ID in auto mode**
   - if unambiguous heuristic exists (e.g., `comment_` prefix), use it
   - otherwise fail with guidance to pass `--type`

Parsing and precedence rules:

1. If `TARGET` is a URL, parse `target_type` from URL shape (comment fragment -> comment, otherwise post).
2. If `--type` is explicitly provided and conflicts with URL-derived type, fail with a clear mismatch error.
3. Accept only Moltbook web URLs (`www.moltbook.com` or `moltbook.com`) for URL parsing; otherwise treat as invalid target input.
4. Trim surrounding whitespace from `TARGET` before parsing.

This ensures support for “id or moltbook.com uri, or whatever else is necessary” while remaining deterministic.

## 8.3 Command flow

1. Resolve target handle using shared `resolve_target_handle` helper.
2. Load agent config and verify API key exists.
3. Parse target into `(target_type, target_id)`.
4. Execute upvote via `PostService`.
5. Print success panel/lines with:
   - resolved target type + ID
   - API message when present
   - optional suggestion metadata if returned

## 8.4 Error handling

- Invalid/ambiguous target -> clear actionable message.
- URL/type mismatch (for explicit `--type`) -> explicit validation error explaining expected `--type`.
- API failures -> existing `MoltbookAPIError` handling style (`message` + optional `hint`).
- Maintain consistent non-zero exit on failures (`ctx.exit(1)`).

## 14) Non-Goals

1. No automated downvote behavior.
2. No new `downvote` CLI command.
3. No broader feed-voting automation outside acted items.
4. No expansion to non-post/comment vote targets.

---

## 15) Acceptance Criteria

Implementation is complete when all are true:

1. Action-stage schema includes `upvote` boolean.
2. `ACTION_SYSTEM_PROMPT` contract and parsing support `upvote`.
3. Automation executes upvote only for autonomously acted queue item and only when `upvote=true`.
4. Automation never auto-downvotes.
5. Dry mode performs zero vote write calls.
6. Upvote API failures do not discard already-completed action writes.
7. `automation setup` behavior prompt copy mentions posting/commenting behavior and upvote circumstances.
8. New `moltbook upvote` command exists with `--handle` and supports both post and comment upvotes.
9. `moltbook upvote` accepts Moltbook URLs and raw IDs (with deterministic disambiguation rules).
10. Tests cover structured output, automation vote behavior, and CLI parsing/execution paths.
11. `README.md` and `docs/AUTOMATION.md` reflect final behavior.
