# AUTOMATION LIST COMMAND SPECIFICATION

## Overview
The `list` command allows the user to inspect the state of the local automation queue. It queries the SQLite database (`automation.db`) and can either show a single status slice or, by default, show **all items grouped by status**.

This command is read-only and intentionally independent from LLM execution prerequisites. It should work even if global provider credentials in `client.json` are not yet configured.

Targeting semantics:

- The command must support `--handle`.
- If `--handle` is provided, use it directly and skip session lookup.
- Without `--handle`, resolve session `active_agent` from `.sessions/<PPID>.json`.
- If session `active_agent` is missing, lazily initialize from `client.json:last_active_agent`.
- If a user always supplies `--handle`, no session file is created.
- `--handle` never mutates session `active_agent` and never mutates `last_active_agent`.

## 1. CLI Command (`commands/automation/list_command.py`)

**Command Signature:**
```python
@click.command(name="list")
@click.option(
    "--handle",
    type=str,
    default=None,
    help="Agent handle to inspect. Defaults to session active agent.",
)
@click.option(
    "--status",
    type=click.Choice(["all", "pending-analysis", "pending-action", "acted"]),
    default="all",
    help="Filter items by their processing status."
)
@click.option(
    "--limit",
    default=None,
    type=click.IntRange(min=1),
    help="Maximum number of items to display. Defaults to all items."
)
def list_command(handle: str | None, status: str, limit: int | None):
    ...
```

**Important:** Do not add short aliases (`-s`, `-n`). Keep options explicit.

**Status Definitions & SQL Mapping:**
The `--status` flag maps directly to SQL `WHERE` clauses in `persistence/automation_store.py`:

| Status Flag (`--status`) | Description | SQL Logic |
| :--- | :--- | :--- |
| **`all`** | Show all groups in ordered sections in one run. | Execute all three slices in order: pending-analysis, pending-action, acted |
| **`pending-analysis`** | Items found by search but not yet checked by LLM. | `WHERE analyzed = 0` |
| **`pending-action`** | Items deemed relevant by LLM, waiting for a reply. | `WHERE analyzed = 1 AND is_relevant = 1 AND replied_item_id IS NULL` |
| **`acted`** | Items that have been successfully replied to. | `WHERE replied_item_id IS NOT NULL` |

For `--status all`, render groups in this exact order:
1. `pending-analysis`
2. `pending-action`
3. `acted`

## 2. Display Logic & Formatting
**Requirement:** The output must visually resemble the Search Results display (dense, readable information) but **must not use Panels**.

**Layout Strategy:**
Use `rich.console.Console.print()` directly with styled text. Separate items with a simple `Rule` or a blank line to keep it clean.

**First line requirement:**
The very first output line should hint users to run help:

```
[dim]Tip: Run '{CLI_NAME} automation list --help' for more options.[/dim]
```

**Visual Template (Per Item):**
```
[bold cyan]{item_id}[/bold cyan] [dim]({item_type})[/dim]
[magenta]@{submolt_name}[/magenta] • [dim]Detected {created_at_relative}[/dim]
[green]Status: {status_label}[/green]
```

**Grouped Template (`--status all`):**
```
[bold]Pending Analysis[/bold]
{items...}

[rule]
[bold]Pending Action[/bold]
{items...}

{items...}
```

## 3. Implementation Steps
1.  **Update Store:** Ensure `persistence/automation_store.py` has a `list_items(handle, status_filter, limit)` method that supports `limit=None` for no limit and applies the correct SQL `WHERE` clauses.
2.  **Service Layer:** Add pass-through method in `services/automation_service.py`.
3.  **Command Handler:**
    - Add `--handle` option.
    - Resolve target handle with session-aware fallback:
      1. explicit `--handle`
      2. session `active_agent` from `.sessions/<PPID>.json`
      3. lazy initialization from `client.json:last_active_agent`
    - Sweep stale `.sessions/<PPID>.json` files during session lookup.
    - If neither a session target nor remembered `last_active_agent` exists, print actionable guidance to run `moltbook agents` or pass `--handle`.
    - Ensure explicit `--handle` path does not create/modify session files.
    - Accept `--status` (default `all`) and `--limit` (default `None`) without short aliases.
    - Print help hint line first.
    - If `status == "all"`, fetch and render each status section in order: pending-analysis, pending-action, acted.
    - If a specific status is selected, render only that group.
    - If no items exist for the selected scope:
      - For `--status all`, print: `[yellow]No items found across all status types.[/yellow]`
      - For a specific status, print: `[yellow]No items found with status '{status}'.[/yellow]`
    - Iterate and print using the "No Panel" layout defined above.
    - Use `rich.rule.Rule` for clean separation between items and groups.
